System Architecture

Component Stack

  1. slack_bot.py - Slack integration layer
    • Handles Socket Mode WebSocket connection
    • Event handlers for app_mention, DMs, slash commands
    • Creates background threads for agent communication
    • Updates Slack messages every 2 seconds with progress
    • Filters ANSI codes and Claude UI elements
  2. agent_manager.py - Process lifecycle manager
    • Manages wrapper process via PTY (pseudo-terminal)
    • Separate thread for continuous output monitoring
    • Thread-safe buffer with locks
    • Detects ---RESPONSE-COMPLETE--- marker
    • 120-second timeout handling
  3. claude_persistent_wrapper.py - Critical bridge
    • Reads user messages from stdin (line-by-line)
    • Calls: claude --print --dangerously-skip-permissions
    • Uses --continue flag after first message
    • Outputs response to stdout
    • Prints ---RESPONSE-COMPLETE--- when done
    • Maintains Claude context across messages
  4. Claude Code CLI - AI engine
    • Binary: /home/appsforte/.nvm/versions/node/v24.10.0/bin/claude
    • Model: Sonnet 4.5 (Claude Max)
    • Full tool access (Bash, Read, Write, Grep, WebSearch, etc.)
    • Processes requests and returns responses

Threading Model

  • Main Thread: Handles Slack events via Socket Mode
  • Background Thread (per request): Calls agent_manager.send_input(), waits for response
  • Monitor Thread (persistent): Continuously reads wrapper PTY output
  • Update Loop (per request): Polls buffer every 2s, updates Slack message

Key Technical Decisions

Why PTY Instead of Direct Pipes?

Pseudo-terminal provides proper interactive communication. Standard pipes would buffer differently and lose real-time capability.

Why Wrapper Instead of Direct Claude Call?

Claude Code's interactive mode doesn't auto-process stdin. The wrapper uses --print for one-shot execution while maintaining context via --continue.

Why Marker System?

Time-based completion detection is unreliable. ---RESPONSE-COMPLETE--- marker guarantees we have full response.

Why 2-Second Updates?

Balance between real-time feel and API rate limits. Too frequent causes API errors, too slow feels unresponsive.

Why Thread-Safe Buffer?

Monitor thread writes to buffer, update loop reads from it. Without locks, race conditions cause missing/duplicate output.
